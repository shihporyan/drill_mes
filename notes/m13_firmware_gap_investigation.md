# M13 韌體時間間隔問題 — 調查結果

**調查日期：** 2026-04-13

## 現象

M13 的 Drive.Log 在特定日期出現兩種異常，M14 從未發生：

1. **時間跳躍**：連續 28 行（每秒一筆）後，時間戳突然跳過 3~495 秒（最常見 14 秒）
2. **時間倒退**：韌體偶爾在正常紀錄中插入一筆「~6 分鐘後」的預讀 row，然後跳回繼續寫

## 各日期狀況

| 日期 | 行數 | Gap 數 | 倒退數 | 備註 |
|------|------|--------|--------|------|
| M13 4/4 | 86,400 | 0 | 0 | 完美 |
| M13 4/5 | 86,399 | 0 | 0 | 完美 |
| M13 4/6 | 83,262 | 355 | 119 | 輕微 |
| M13 4/7 | 64,254 | 2,046 | 259 | 嚴重 |
| M13 4/8 | 70,678 | 1,873 | 295 | 嚴重 |
| M13 4/10 | 58,183 | 2,774 | 423 | 最嚴重 |
| M14 所有日期 | 86,400 | 0 | 0 | 永遠完美 |

## 倒退模式（以 10Drive.Log line 232-234 為例）

```
line 232: 23:59:42  counter=150535003  ← 正常
line 233: 00:05:52  counter=150535137  ← 跳到 6 分鐘後（預讀）
line 234: 23:59:57  counter=150535008  ← 跳回正常位置
```

Counter 值證實預讀 row 是真實的未來資料（counter 持續遞增），不是損壞。

## 結論

確認為 **M13 韌體行為**（間��性），非資料遺失。原因推測為���體的 dual-buffer 或 write-ahead 機制。M14 韌體版本可能不同。

## 修復

Parser 從 row-counting（`+1 per row`）改為 timestamp-delta counting：
- 排序 + 去重（修正倒退）
- 計算相鄰 row 的時間差，歸入當前 state
- 跨小時邊界自動分配
- Gap cap = 120 秒（超過不計）

修復效果：M13 bad day 從 64,254 秒 → 82,151 秒（+28%），接近理論 86,400 秒。
