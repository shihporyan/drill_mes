> 日期：2026/03/17  
> 機台：DRILL-01（Takeuchi 鑽孔機）  
> 主機 IP：192.168.1.2  
> Log 路徑：`\\192.168.1.2\log\`（平鋪結構，以日期前綴區分）

## 一、專案目標

| # | 目標 | 說明 |
|---|------|------|
| 1 | **稼動率** | 計算鑽孔機實際鑽孔時間 vs 閒置時間 |
| 2 | **自動排程** | 推算程式鑽孔時間，支援排程決策 |

為達成以上目標，需要從 log 取得：

- **[A] 機台狀態**：正在鑽孔（RUN）還是閒置（RESET/STOP）
- **[B] 執行程式**：正在鑽哪個 NC 程式 → 對應工單號 → 知道孔徑和孔數
- **[C] 板號追蹤**：正在鑽哪一塊板（A~F），此項需進一步驗證

## 二、Log 檔案結構

鑽孔主機 `\\192.168.1.2\log\` 為平鋪結構（無子資料夾），以日期前綴命名（帶前導零）：

```
01Alarm.Log, 01Drive.Log, 01FILE.Log, ...
17Alarm.Log, 17Drive.Log, 17FILE.Log, ...
```

每天產生 6 個 log 檔，以 17 號為例：

| 檔案 | 大小 | 行數 | 編碼 | 用途 |
|------|------|------|------|------|
| 17Drive.Log | 15 MB | 54K | ASCII | 每秒狀態記錄（核心）|
| 17FILE.Log | 730 KB | 55K | ASCII | NC 程式完整內容 |
| 17MACRO.Log | 2.5 MB | 21K | Shift_JIS (CP932) | 系統內部除錯 |
| 17TARN.Log | 112 KB | 1.6K | Shift_JIS (CP932) | 起動/停止/換刀事件 |
| 17TX1.Log | 55 KB | 644 | Shift_JIS (CP932) | 操作員操作記錄 |
| 17Alarm.Log | 266 B | 7 | ASCII | 報警記錄 |

## 三、各 Log 欄位解析

### Drive.Log（核心 — 稼動率計算來源）

逐秒記錄，CSV 格式：

```
日期,時間,模式,狀態,程式名,X座標,Y座標,刀具號,針徑,訊息碼,累計值,...
```

| 欄位 | 位置 | 值範例 | 意義 |
|------|------|--------|------|
| 模式 | Col3 | `AUTO` / `MAN` | 自動模式 / 手動模式 |
| 狀態 | Col4 | `RUN` / `RESET` / `STOP` | **鑽孔中 / 閒置 / 暫停** |
| 程式名 | Col5 | `O100.txt` | 當前載入的主程式 |
| 刀具號 | Col8 | `041`, `001`, `096` | 刀座編號 |
| 針徑 | Col9 | `0.150`, `1.100`, `2.100` | 當前鑽頭直徑 (mm) |

### TARN.Log（精確事件時間戳）

離散事件記錄，Shift_JIS 編碼，關鍵事件：

| 事件 | 日文 | MB 碼 | 意義 |
|------|------|-------|------|
| 起動 | 起動 | MB300700 | **開始鑽孔**（精確到毫秒）|
| 停止 | 停止 | MB300721 | **停止鑽孔** |
| 重置 | リセット | MB300740 | 重置狀態 |
| 異常重置 | 異常リセット | MB300940 | 異常後重置 |
| 換刀 | ToolChenge | — | `ST:[刀座號] BLOCK:[區塊號]` |

### TX1.Log（操作記錄 + 程式載入）

Shift_JIS 編碼，關鍵事件：

- `LoadProgram(D:\Takeuchi\NcProgram\O2603044.B)` — 載入生產程式
- `CHANGE SETTING ITEM:[TOOL DIA][No.1] SETTING:[1.100 => 0.150]` — 參數變更
- `BUTTON PUSH SCREEN:[TOUCH PANEL] BUTTON:[START]` — 操作員按下啟動

### FILE.Log（NC 程式完整內容）

每次 LoadProgram 時完整記錄 NC 程式碼。可解析出：

- 子程式結構（O301~O323）
- 每個子程式的鑽孔座標數（= 孔數）
- 刀具呼叫順序（T41, T1, T50, T58, T56...）

### Alarm.Log / MACRO.Log

- Alarm.Log：報警代碼記錄，用於故障分析
- MACRO.Log：系統內部除錯日誌，一般分析不需要

## 六、效率問題討論

### Q1：怎麼最小化 log 存儲？

**結論**：不存完整 Drive.Log（15MB/天），只抽取「狀態變化」事件。54,000 行中實際狀態變化約 90 次。Python 腳本每 10 分鐘抽取新增的狀態變化行，寫入 MES 資料庫。

### Q2：開始/結束時間用哪個 log 最有效率？

**結論**：用 TARN.Log。有明確的 `起動`/`停止`/`リセット` 事件標記，精確到毫秒，不用掃描 54K 行。Drive.Log 當交叉驗證用。

### Q3：可以用 Python 腳本抽取嗎？

**結論**：完全可以，而且應該這樣做。log 格式固定，用正規表達式即可穩定抽取：

- Drive.Log → `split(',')` 取 Col4/Col8/Col9
- TARN.Log → CP932 解碼 + grep 起動/停止/ToolChenge
- TX1.Log → CP932 解碼 + grep LoadProgram
- FILE.Log → 偵測到新 LoadProgram 時才解析孔數

### Q4：每個工號結構不同的難點？

**結論**：三個難點：

1. **O100.txt 是主控程式**，不是生產程式。每次換工號內容會變
2. **子程式編號無固定對應**，O301 不一定是 A 板
3. **無法直接知道當前子程式**，需用刀具號變化序列反查 NC 程式的 T 指令順序

### Q5：跟 MES 資料庫結合

**結論**：MES 回答「誰 + 哪個工單 + 哪台機台」，log 回答「什麼時候開始/結束 + 鑽了多久」。兩邊用時間戳 + 機台編號 JOIN。Log 端專注提供狀態時間，不需要自己辨識工單號。

### Q6：即時儀表板架構

**結論建議**：

```
鑽孔機 log 共享資料夾
    ↓ (每 10 分鐘)
Python 腳本抓取 + 解析狀態變化
    ↓
SQLite / MES 資料庫
    ↓
Flask API
    ↓
前端儀表板（自動刷新）
```

儀表板顯示：目前狀態、程式名、針徑、本日稼動率、預估剩餘時間。
操作員資訊從 MES 資料庫 JOIN。
核心資料表 `machine_events`：每天每台約 100~200 筆。

## 八、Log 分析核心價值

1. **精準時間戳**：機器自動記錄，毫秒精度，防止人為失誤或漏記
2. **程式孔數可知**：FILE.Log 完整記錄 NC 程式，可解析每個子程式的孔數
3. **稼動率可即時計算**：Drive.Log 每秒記錄 RUN/RESET/STOP 狀態

邏輯鏈：
```
LoadProgram → 知道程式名
         → FILE.Log 知道孔數和刀具
TARN.Log  → 精確開始/停止時間
Drive.Log → 每秒狀態（RUN/IDLE）
MES DB    → 工單號、操作員、料號
```
